Executive Summary

For the credit committee.

Build a merchant credit passport as the operating system for Dutch e-commerce SME lending. Combine consent-based data from bank, PSPs, webshops, marketplaces and accounting — feed it through interpretable scoring — and put a human credit officer at the decision point. Start small. Earn the data. Learn the portfolio.

What

A standardised, consent-based underwriting file that reconstructs revenue stability, refunds, chargebacks, platform concentration and cash-flow coverage from the data merchants already produce.

Why now

PSD2 makes bank data routine. Commercial APIs expose PSP, webshop and accounting data. Bookkeeper-first distribution is open in NL. Existing lenders still underwrite on one slice.

The ask

Pre-seed funding to originate the first 5–10 loans under manual policy, build the data substrate, and reach 70 loans of repayment history within 15 months — enough for Seed diligence.

02 · Product

The Merchant Credit Passport.

A standardised, consent-based underwriting file. Built for a human credit officer, not a black box.

  • Consent-based — merchant explicitly connects each data source and can revoke at any time.
  • Reconstructs reality — revenue stability, settlement timing, refunds, chargebacks, platform concentration, supplier punctuality.
  • Interpretable — every feature and deduction is visible. No mystery score.
  • Human-in-the-loop — passport supports a credit officer; it does not replace one.
  • Honest about limits — Ledger improves the signal; it does not remove credit risk.

03 · Underwriting Model

Two engines. Rules decide, ML learns.

Six data sources become about fifteen scored signals. A transparent rules engine turns them into a decision today. A machine-learning model reads the same signals in parallel — logged and studied, but never in charge until it has earned that on real repayments.

The two engines, in code
rules_engine.py Live today 
# Phase 1 — the engine that actually decides
def underwrite(merchant):
    f = compute_features(merchant)   # 15 signals, 6 sources

    # 1 - Hard knockouts: any one means decline
    if knockout(f):                  # sanctions, KvK, UBO,
        return "DECLINE"           # recon gap, history...

    # 2 - Count gates vs green / amber / red bands
    red, amber = count_flags(f)
    if red >= 3:
        return "DECLINE"
    if red == 2 or red + amber >= 5:
        return "MANUAL_REVIEW"

    # 3 - Size the loan (25k Year-1 cap)
    limit = min(merchant.requested,
                f.monthly_cashflow * f.tenor * 0.33,
                f.monthly_gmv * 2.5,
                25_000)
    return "APPROVE", limit
shadow_model.py Shadow · logged 
# Phase 2 — same data, learns but does NOT decide
from sklearn.ensemble import GradientBoostingClassifier

model = GradientBoostingClassifier(
    n_estimators=200, max_depth=4,
    learning_rate=0.05)

model.fit(X_train, y_default)    # y = did the loan default?

# At application time - same merchant, same f:
def shadow_score(merchant):
    f = compute_features(merchant)
    X = to_vector(f)            # 54-feature vector
    p = model.predict_proba(X)[0, 1]
    log(merchant.id, p)          # stored vs real outcome
    return p                      # officer never sees it

Same merchant, same fifteen signals. Engine A returns a decision a credit officer acts on. Engine B returns a probability we simply write down — checked against what actually happens to each loan, until we have enough repaid history to trust it. The model never touches a live decision in Phase 1 or 2.

Same merchant, same six sources, through both engines. Engine A returns a decision the officer acts on. Engine B takes the identical data, builds the 54-feature vector and returns a calibrated probability — here a low 2.5% — which is stored next to the eventual repayment but never shown to the officer. The drivers are the model’s overall weighting, not a per-merchant explanation (that’s Phase 3). Run all four demo merchants live in the next section.

Where this goes — two phases
Now active

Phase 1 · Rules decide

  • Interpretable gates + 7 hard knockouts
  • A credit officer signs off every file
  • Every signal and threshold is visible
Next

Phase 2 · ML learns in shadow

  • Gradient boosting scores in parallel, logged only
  • Checked against real repayment outcomes
  • Earns into decisions only after ~200 loans validate it
The 7 hard knockouts — any one is an automatic decline
Sanctions / PEP hit
KvK inactive
UBO / director mismatch
Bank–PSP recon gap > 20%
Trading history < 24 months
Monthly GMV < €25k
Negative balance > 50% of days

Everything else — refunds, chargebacks, volatility, margin — is a scored gate, not a knockout. Gates colour green / amber / red and feed the counting rule in the code above.

How the limit is sized
Approved limit = the smallest of
requested amount
monthly cashflow × tenor × 0.33
monthly GMV × 2.5
€25,000 Year-1 hard cap
The €25k cap is always enforced In the pilot it usually binds — most approvals land at the €25,000 Year-1 ceiling rather than the cashflow or GMV anchors.

Pricing band: A · 11.0% (clean) · B · 12.5% (default) · C · 14.0% (any red gate or thin coverage).

              A narrow wedge, a credible MVP, defensibility through operational learning.

              Where Ledger fits in the SME-lending landscape, how it makes money, and what the next 24 months look like.

              Business model · six panels

              Target customer

              Dutch e-commerce SMEs with €0.3M–€5M GMV, ≥2 years trading, sales fragmented across PSPs, marketplaces and webshops.

              Revenue model

              • Net interest margin on the portfolio
              • 2.0% origination fee at draw-down
              • Small late / early-repayment fees (capped)

              Initial product

              • Amortising business term loan
              • €10,000 – €150,000
              • 3 – 18 months
              • No asset collateral in the base product

              Why now

              PSD2 and consent-based data access make broader underwriting commercially viable. Bookkeeper-first distribution is open in NL.

              Why Ledger can start

              • Narrow customer wedge (NL e-commerce)
              • E-commerce-specific risk features
              • Bookkeeper referral channel
              • Front-end-heavy MVP buildable today

              What this is not

              Not a licensed bank. The MVP is not a real underwriting engine. No claim of instant approval, permanent moat, or removal of credit risk.
